Part 1: Requirements Engineering
1.1 Stakeholder Analysis
Although the main users of the system are people who use the software and collaborate on documents, the platform affects many other stakeholders throughout its lifecycle. Since this project involves real-time collaboration, AI integration, authentication, and document management, it is important to consider stakeholders beyond just the end users. Below are four key stakeholder groups and how they influence the system’s design.
DevOps and Operations Engineers
These are the people responsible for deploying, running, and maintaining the system once it is built. Their main concern is making sure the platform is stable, scalable, and reliable.

Goals
Keep the system available and responsive.
Ensure the system can scale as more users join.
Monitor performance and detect issues quickly.
Support smooth deployments and rollbacks.
Concerns
Real-time collaboration can be complex and resource-intensive.
AI services may fail or respond slowly.
Data loss during crashes or outages.
Secure storage of API keys and credentials.
Impact on the System
Because of them, we need clear monitoring, logging, and backup strategies. The architecture must support horizontal scaling and graceful degradation. For example, if the AI service goes down, document editing should still work as normal. Their needs strongly influence our non-functional requirements such as availability, scalability, and reliability.
Security and Compliance Stakeholders
This group includes security engineers and anyone responsible for privacy and regulatory compliance. Since documents may contain sensitive information and AI services process user content, security is a major concern.

Goals
Protect document data from unauthorized access.
Ensure secure authentication and authorization.
Prevent data leaks when communicating with third-party AI providers.
Maintain user trust.
Concerns
Sending document content to external LLM APIs.
Encrypting data both in transit and at rest.
Proper role-based access control (owner, editor, viewer, etc.).
Logging and storing AI interaction history securely.
Impact on the System
Their concerns require strong authentication and authorization mechanisms, encrypted communication, and clear policies on what content is shared with AI services. They also influence how permissions are modeled and how version history and audit logs are handled.
External AI Service Providers
Since AI functionality is a core feature of the platform, we rely heavily on third-party large language model (LLM) providers to power capabilities for our software, such as rewriting, summarization, translation, and restructuring. Thus, these providers directly impact the quality, performance, and cost of our platform.
Goals
Ensure proper API usage within rate limits and token limits.
Maintain stable integration through well-defined contracts.
Concerns
Rate limiting and quota restrictions.
Token limits for long documents.
Latency variability.
Model changes or pricing updates.
Impact on the System
This dependency forces us to carefully design the AI integration. AI requests may need to be asynchronous, and the system must handle cases where responses are delayed or fail. We also need abstraction layers so we are not tightly coupled to one provider. Cost management strategies (such as user usage limits) are also influenced by this stakeholder.
Product Owners and Business Stakeholders
This group includes founders, product managers, and investors who care about the platform's long-term success.
Goals
Deliver a competitive collaborative editor with strong AI features.
Attract and retain users.
Control operational costs, especially AI-related expenses.
Define a sustainable monetization model.
Concerns
High API costs from frequent AI usage.
Scalability as the user base grows.
Balancing feature richness with usability.
Time-to-market constraints.
Impact on the System
Business stakeholders influence feature prioritization, pricing strategies, and cost-control mechanisms such as AI quotas or tier-based access. They also impact architectural decisions that affect scalability and long-term maintainability.

Definition
FR-COL-01: Simultaneous Editing
Description: The system shall support concurrent editing of the same document by multiple users.
Trigger: Two or more authenticated users open the same document.
Expected Behavior: Edits made by any user are propagated to all other connected users in near real time.
Acceptance Criteria:
Text updates appear to other users within 500 milliseconds under normal load.

Description: The system shall display active collaborators, optionally along with their cursor positions.
Trigger: A user joins an active document session.
Expected Behavior: The user list updates dynamically and displays participant presence.
Acceptance Criteria:
Active participants are visible to all users in the session.
Cursor positions are visually distinguishable.

Description: The system shall prevent data loss during simultaneous edits to overlapping text regions.
Trigger: Two users edit overlapping sections at the same time.
Expected Behavior: A defined conflict-resolution strategy merges updates without silently overwriting text.
Acceptance Criteria:
No user’s input is permanently lost.
Conflicting edits are handled consistently and predictably.

Description: The system shall synchronize the complete current document state when a user joins an active editing session.

The loaded document matches the authoritative server state.
No partial or inconsistent content is displayed.
The user can begin editing immediately after synchronization.

Description: The system shall handle temporary client disconnections without permanent data loss.
Trigger: A user loses network connectivity during editing.

Local edits are buffered.

Conflicts are resolved using the defined merge strategy.

No confirmed edits are permanently lost.
The user is notified of the reconnection status.
Document state becomes consistent after reconnection.

Changes can be traced to a specific user.
Attribution data is available to version history or UI components.

Acceptance Criteria:
Overlapping edits are visually indicated.
Users are not silently overridden.

Requirement ID
Definition

Description: The system shall allow users to request AI assistance on selected text.
Trigger: A user selects text and chooses an AI action (e.g., summarize, rewrite, translate).
Expected Behavior: The selected text and request type are sent to the backend AI service.
Acceptance Criteria:
The AI request is initiated without reloading the page.

FR-AI-02: Suggestion Presentation
Description: The system shall present AI-generated suggestions in a non-destructive manner.
Trigger: The AI returns a response.
Expected Behavior: The suggestion appears as a preview (e.g., side panel or tracked-change style).
Acceptance Criteria:
The original text remains unchanged until the user explicitly accepts the suggestion.
The user can accept or reject the suggestion.
FR-AI-03: Partial Acceptance
Description: The system shall allow users to partially edit or modify AI-generated suggestions before applying them.
Trigger: The user edits the AI-proposed output before acceptance.
Expected Behavior: The system treats the edited suggestion as user-generated content upon approval.
Acceptance Criteria:
Modified suggestions replace only selected text.
The action is reversible via undo functionality.
FR-AI-04: Context Scope
Description: The system shall limit the text sent to the AI service to the user-selected text and, optionally, a predefined surrounding context window (e.g., the same paragraph), without including the entire document unless explicitly required by the feature.
Trigger: A user selects text and invokes an AI action (e.g., summarize, rewrite, translate).
Expected Behavior:
The backend constructs an AI request containing:
The selected text.
Only the configured contextual window (if applicable).
The requested AI action type.
No unrelated document content is included in the request payload.
Acceptance Criteria:
The AI request payload contains only the selected text plus the allowed context window.
Content outside the defined scope is never transmitted to the AI service.
For large documents, request size remains within predefined limits (e.g., max token threshold).
Scope behavior is consistent across all AI features.
FR-AI-05: AI Failure Handling
Description: The system shall gracefully handle AI request failures, including timeouts, service unavailability, or API errors, without modifying the document.
Trigger: An AI request times out, returns an error response, or the AI service is unavailable.
Expected Behavior:
The system displays a clear and user-friendly error message.
The original document content remains unchanged.
The user is given the option to retry the request.
Acceptance Criteria:
No document content is altered when an AI request fails.
An error notification is displayed within 2 seconds of failure detection.
A retry option is available without requiring page reload.
The failure is logged for monitoring purposes.
FR-AI-06: AI Suggestion vs Concurrent Edits
Description: The system shall detect changes to the selected text that occur while an AI request is being processed and prevent automatic application of outdated suggestions.
Trigger: A user invokes AI on selected text, and the selected region is modified (by the same user or another collaborator) before the AI response is returned.
Expected Behavior:
The system compares the current text with the original text used in the AI request.
If a mismatch is detected, the system prompts the user with options:
Re-run the AI on the updated text
Apply the suggestion anyway
Discard the suggestion
Acceptance Criteria:
AI suggestions are never silently applied to modified text.
A mismatch detection mechanism prevents automatic overwriting.
The user must explicitly confirm before applying a potentially outdated suggestion.
No collaborator’s edits are overwritten without confirmation.
FR-AI-07: AI Interaction Record
Description: The system shall record metadata for each AI interaction to support traceability and future reference.
Trigger: An AI suggestion is generated, accepted, rejected, or modified.
Expected Behavior:
 The system stores metadata including:
AI feature type (e.g., summarize, rewrite)
Timestamp
Reference to the affected text segment (e.g., anchor or hash)
Outcome status (accepted, rejected, partially applied)
Acceptance Criteria:
AI interactions are stored and linked to the corresponding document.
Users can view a list of past AI interactions for a document.
Each record includes feature type, timestamp, and outcome status.
Stored metadata does not include unnecessary full-document content.
FR-AI-08: AI Permission Enforcement
Description: The system shall enforce role-based permissions for invoking AI features within a document.
Trigger: A user attempts to invoke an AI feature.
Expected Behavior:
The system verifies the user’s role for the document.
Users with insufficient permissions (e.g., viewers) are prevented from invoking AI features.
Authorized users (e.g., editors, owners) may invoke AI normally.
Acceptance Criteria:
Users with a viewer role cannot initiate AI requests.
The AI action controls are disabled or hidden for unauthorized users.
If an unauthorized request is attempted via API manipulation, the backend rejects it with an appropriate error response.
Authorized roles can invoke AI without restriction.

1.3 Non-Functional Requirements
	Latency
NFR-LAT-01
Under normal operating conditions, 95% of text edits made by one active collaborator shall be visible to other collaborators editing the same document within 500 ms of submission.
Justification: Delays above this threshold make real-time collaboration feel inconsistent and can cause users to type over stale content.
NFR-LAT-02
For AI assistant requests on selections up to 2,000 characters, the system shall acknowledge the request and display a visible processing state within 500 ms, and shall begin streaming or returning the AI-generated response within 10 seconds in 95% of requests.
Justification: Users need immediate feedback that the request was received, while a short wait for AI generation remains acceptable if the system clearly indicates progress.
NFR-LAT-03
For documents under 10,000 characters, the editor shall load the document content and become ready for user editing within 2 seconds in 95% of requests under normal operating conditions.
Justification: Longer waits interrupt the writing flow and make opening shared documents feel unreliable.


Scalability
NFR-SCALE-01
The system shall support at least 10 concurrent active editors per document, where each user performs up to 5 edit operations per second, while maintaining edit propagation latency within 500 ms (95th percentile).
Justification: Ensures real-time collaboration remains responsive under realistic multi-user editing loads.
NFR-SCALE-02
The system shall support at least 1,000 concurrently active documents, each with at least 2 active users, without exceeding 80% average CPU utilization on application servers.
Justification: Prevents performance degradation during peak usage across the platform.
NFR-SCALE-03
All backend services handling document editing and AI requests shall be horizontally scalable, with no reliance on in-memory session state, enabling linear scaling by adding instances behind a load balancer.
Verification: System can scale from 1 → N instances with no functional changes and ≤10% performance degradation.
NFR-SCALE-04
The system architecture shall support projected workload growth of at least 1.5x in monthly active users and active documents over 12 months without requiring changes to public API contracts or replacement of the core real-time synchronization mechanism.
Justification: The assignment explicitly asks for an expected growth model, and this requirement ensures the design can evolve without costly architectural rewrites.


	Availability
NFR-AVAIL-01
The system shall maintain ≥99% monthly uptime, excluding scheduled maintenance windows not exceeding 10 hours per month.
Verification: Uptime measured via external monitoring over rolling 7-day periods.
NFR-AVAIL-02
In the event of AI service failure or timeout (>5 seconds), the system shall gracefully degrade, allowing document editing, saving, and collaboration features to remain fully operational, with user-visible notification of AI unavailability.
Justification: Prevents external dependency failure from blocking core functionality.
NFR-AVAIL-03
The system shall automatically reconnect and restore editing sessions within 30 seconds after network recovery, preserving unsynchronized local changes and resolving conflicts using operational transformation or CRDT mechanisms.
Verification: No data loss in simulated disconnection scenarios up to 30 seconds.


	Security and Privacy
NFR-SEC-01
All client-server and service-to-service communication shall be encrypted using TLS 1.2 or higher with strong cipher suites.
Verification: No endpoints accessible over unencrypted HTTP.
NFR-SEC-02
All document content and metadata shall be encrypted at rest using AES-256 or equivalent, including backups and replicas.
Verification: Storage-level encryption enabled and enforced.
NFR-SEC-03
AI interaction logs (prompts and responses) shall be access-controlled via role-based access control (RBAC) and retained for a maximum of 30 days, after which they are automatically deleted or anonymized.
Justification: Balances debugging needs with privacy protection.
NFR-SEC-04
The system shall explicitly notify users before transmitting any document content to third-party AI services and shall allow users to opt out of AI processing per document or request.
Verification: Consent UI present and enforced before API calls.

Part 2: System Architecture
2.1 Architectural Drivers
The architectural drivers are prioritized based on their impact on user experience and system feasibility.
Low-latency collaboration is ranked highest because real-time feedback is essential to usability—delays beyond 500 ms significantly degrade the collaborative experience. Consistency is handled using eventual consistency models (e.g., CRDT/OT), accepting temporary divergence in favor of responsiveness. Availability and scalability ensure the system remains usable under failure and growth conditions, while AI integration is treated as a core but non-blocking feature to avoid disrupting editing workflows.


No.
Driver
Related Requirements
Description
Impact on Architecture
1
Low-Latency Real-Time Collaboration
NFR-LAT-01, FR-COL-01
The system must propagate edits between users within 500 ms to enable seamless real-time collaboration.
Forces a push-based communication model (e.g., WebSockets) over polling. Requires a dedicated real-time collaboration service maintaining active sessions and in-memory state. Introduces challenges in horizontal scaling (connection affinity, session distribution) and requires efficient event broadcasting mechanisms. Trade-off: increased infrastructure complexity vs superior user experience.
2
Consistency & Conflict Resolution
FR-COL-03, NFR-AVAIL-03
The system must ensure concurrent edits do not result in data loss or inconsistent document state.
Requires adoption of conflict resolution algorithms (CRDT or OT) and a structured document model. CRDT enables offline editing and eventual consistency but increases memory and implementation complexity; OT simplifies transformation logic but typically requires centralized coordination. Trade-off: strong consistency vs scalability and offline capability.
3
Availability & Graceful Degradation
NFR-AVAIL-02, NFR-AVAIL-03
The system must remain functional during partial failures, especially when AI services are unavailable.
Requires decoupled architecture separating core editing from AI services. Introduces fallback mechanisms (disable AI features while preserving editing), retry strategies, and local state buffering for reconnection. Trade-off: additional system complexity vs resilience and uninterrupted user workflows.
4
Scalability
NFR-SCALE-01 to NFR-SCALE-04
The system must support increasing numbers of users, documents, and concurrent editing sessions.
Requires stateless API services, horizontal scaling, and load balancing. The real-time layer must support distributed event propagation (e.g., pub/sub systems). Trade-off: distributed system complexity vs ability to handle growth and high concurrency.
5
Security & Access Control
NFR-SEC-01 to NFR-SEC-04, FR-USER-02
The system must protect sensitive document data and enforce role-based access control.
Requires authentication mechanisms, role-based access control (RBAC), and encryption in transit and at rest. AI integration must include explicit user consent and data handling policies when interacting with third-party services. Trade-off: stricter security measures may increase latency and system complexity.
6
AI Integration & Cost Control
FR-AI-01 to FR-AI-08, NFR-LAT-02
AI features must be responsive, cost-efficient, and seamlessly integrated without disrupting editing.
Requires asynchronous processing model (e.g., job queues or event-driven workflows) to avoid blocking the editor. Introduces AI service abstraction layer, prompt templating system, and context selection strategies (e.g., partial document vs full document). Requires rate limiting, quotas, or budgeting mechanisms. Trade-off: richer AI capabilities vs latency, cost, and system complexity.
7
Usability & User Experience

2.2.4 Feature Decomposition
The system is divided into key modules:
Frontend: User interface and editor
Real-Time Service: Handles live collaboration
Backend API: Manages logic and requests
AI Service: Processes AI features
Database: Stores documents, users, and versions
This modular design allows independent development and scalability.
2.2.5 AI Integration Design
Only selected text is sent to the AI to reduce cost and protect privacy
AI results are shown as suggestions, not automatically applied
Users can accept, reject, or modify AI output
AI requests are handled separately from core editing to avoid system disruption
2.2.6 API Design
The system exposes APIs for:
Documents: create, read, update, delete
AI Features: summarize, rewrite, translate
Users: login, register, manage roles
AI requests may be asynchronous, with clear success or error responses.
2.2.7 Authentication and Authorization
The system uses role-based access control (owner, editor, commenter, viewer). Authentication ensures only authorized users access documents, while authorization controls actions like editing, sharing, and using AI features.
2.2.8 Communication Model
The system uses real-time communication for collaboration. Users receive updates instantly. If a connection is lost, changes are buffered and synchronized upon reconnection

2.3 Code Structure & Repository Organization
This section describes how the codebase is organized to support modular development, scalability, and team collaboration.
Repository Strategy
The project will use a monorepo structure, where frontend, backend, and shared code are maintained in a single repository.
Justification:
Easier collaboration for a small team
Simplifies dependency management
Allows shared types and models between frontend and backend
Easier to maintain consistency between API and UI

Directory Structure
The repository is organized as follows:
/project-root
│
├── frontend/        # Web application (UI, editor, components)
├── backend/         # API, business logic, services
├── realtime/        # Real-time collaboration service
├── ai-service/      # AI integration and prompt handling
├── shared/          # Shared types, constants, utilities
├── docs/            # Diagrams and documentation
└── README.md        # Setup and usage instructions
Explanation:
frontend/ handles the user interface and editor logic
backend/ contains API endpoints, authentication, and document management
realtime/ manages live collaboration and synchronization
ai-service/ handles communication with external AI providers
shared/ avoids duplication by storing common code

Shared Code
Shared code (e.g., data models, validation rules, constants) is placed in the shared/ directory.
Purpose:
Ensure consistency between frontend and backend
Avoid duplicate definitions (e.g., document schema, user roles)
Improve maintainability

Configuration Management
Sensitive data such as API keys, database credentials, and AI provider tokens are stored in environment variables.
Approach:
Use .env files for local development
Never store secrets in the repository
Use environment-based configuration for different environments

Testing Strategy
The system will include multiple levels of testing:
Unit Tests: Test individual components and functions
Integration Tests: Test interaction between services (e.g., API + database)
End-to-End Tests: Test complete workflows (e.g., create and edit document)
For AI features, mock responses can be used to avoid repeated API calls during testing.
Summary
The chosen repository structure supports modular development, reduces duplication through shared code, and enables efficient collaboration within the team. The monorepo approach aligns well with the project scope and team size.
